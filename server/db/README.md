# Database & Migrations

This directory holds the Praetor server's database layer. The historical Drizzle migration is complete: every runtime table is modeled in `schema/*.ts`, every repository in `server/repositories/` uses Drizzle, and every schema change goes through Drizzle Kit.

## Layout

- `schema/*.ts` — Drizzle table definitions. The single source of truth for the live schema. `schema/index.ts` re-exports each table.
- `migrations/` — Drizzle Kit migration files plus a `meta/` snapshot directory. Generated by `drizzle-kit`, applied by the programmatic migrator in `scripts/run-migrations.ts`.
- `drizzle.ts` — exports the `db` instance, the `DbExecutor` type used by repositories, and the `withDbTransaction` / `executeRows` helpers.
- `index.ts` — exports the underlying `pg` Pool (for the demo-seed bootstrap) plus the `query` and `buildBulkInsertPlaceholders` helpers used only by `demoSeed.ts` / `scripts/seed-demo.ts`. Application code goes through Drizzle.
- `schema.sql` — frozen historical baseline (~2,100 lines). Authoritative for any DB bootstrapped before Drizzle Kit took over. Don't edit; new schema changes belong in `schema/*.ts` + a generated migration.
- `add_*.ts`, `make_*.ts`, `remove_*.ts` — frozen pre-Drizzle ad-hoc migration scripts. Don't edit, don't extend the pattern.
- `bootstrapAdmin.ts`, `demoSeed.ts`, `demoSeedManifest.ts`, `apply_migration_associations.ts` — seed/bootstrap helpers, not migrations.

## Authoring a schema change

```bash
# 1. Edit server/db/schema/<table>.ts (or add a new file and re-export from schema/index.ts).
cd server

# 2. Generate the SQL migration from the schema diff:
bun run db:generate --name add_clients_priority

# 3. Review the generated SQL in db/migrations/0NNN_*.sql.
#    Hand-edit if needed — see "Idempotent guards" below.

git add db/schema/ db/migrations/
git commit
```

For changes that don't touch the TS schema (rare — usually a one-off data migration), use:

```bash
bun run db:generate:custom --name backfill_legacy_clients
# Produces an empty 0NNN_*.sql; write the raw SQL by hand.
```

Additive changes (new column, new index) are safe to ship as-generated. **Destructive changes** (drop column, change column type, tighten a constraint) need extra care — read the generated SQL and add `USING` clauses, backfill statements, or hand-edited guards as the change requires.

## Idempotent guards

Some live DBs were bootstrapped from `schema.sql` (which already defines the complete pre-Drizzle baseline) and then had Drizzle migrations layered on top. When a migration adds a constraint or index that *already exists* under a different (Postgres-auto-generated) name, the raw `ALTER TABLE ADD CONSTRAINT` will collide.

The pattern, used heavily in `0018_model_remaining_constraints_and_fks.sql`:

```sql
DO $$ BEGIN
    -- Drop any pre-existing auto-named version, then add the canonical Drizzle-named one.
    ALTER TABLE "users" DROP CONSTRAINT IF EXISTS "users_role_fkey";
    IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname = 'users_role_roles_id_fk') THEN
        ALTER TABLE "users" ADD CONSTRAINT "users_role_roles_id_fk"
            FOREIGN KEY ("role") REFERENCES "public"."roles"("id") ON DELETE RESTRICT;
    END IF;
END $$;
```

Use the same pattern for indexes (`CREATE INDEX IF NOT EXISTS`) and tables that exist in `schema.sql` but are being newly added to `schema/*.ts` (`CREATE TABLE IF NOT EXISTS`).

**Once committed, a migration is hashed by Drizzle Kit and recorded in `__drizzle_migrations`. Don't edit it after merge** — even comment changes break the hash on already-applied DBs.

## Applying migrations

```bash
cd server
bun run db:migrate
```

Idempotent: re-running is a no-op if everything's applied. The runner is `scripts/run-migrations.ts` (calling `drizzle-orm/node-postgres/migrator` directly), not `drizzle-kit migrate` — see "Why not `drizzle-kit migrate`" below.

## Local DB setup

Bootstrap order for a fresh local DB:

1. Apply `schema.sql` (`psql -f db/schema.sql` or via the existing Docker bootstrap).
2. From `server/`, run `bun run db:migrate`. The migrations are written with `IF NOT EXISTS` guards so they're no-ops on a DB that already matches; their rows still land in `__drizzle_migrations` so subsequent migrations apply normally.

For a Drizzle-only fresh DB (skipping `schema.sql`): every existing migration uses `CREATE TABLE IF NOT EXISTS` and friends, so applying them in order produces a fully-formed schema. This path isn't the production bootstrap, but it's how `db:check` and CI verify the migrations.

## Available scripts

From `server/`:

| Script | Purpose |
|---|---|
| `bun run db:generate` | Generate a migration from the diff between `schema/*.ts` and the latest snapshot. |
| `bun run db:generate:custom` | Generate an empty migration file for hand-written SQL. |
| `bun run db:migrate` | Apply pending migrations to the DB pointed to by `DB_*` env vars. |
| `bun run db:check` | Verify that snapshots and migration SQL are consistent (run in CI to catch drift). |
| `bun run db:studio` | Open Drizzle Studio (browser-based DB explorer). **Local dev only — never point at prod credentials.** |

### Why not `drizzle-kit migrate`?

drizzle-kit 0.31's `migrate` CLI wraps the migrator in a hanji `renderWithTask` spinner that swallows error output in non-TTY environments (CI, deploy hooks, anything where stdout isn't a real terminal). When migrations fail, the CLI exits 1 with no error message — the actual exception never reaches stderr because the spinner consumes the rejected promise.

We invoke `drizzle-orm/node-postgres/migrator` directly via `scripts/run-migrations.ts` instead. The migration files, journal, snapshot, and `__drizzle_migrations` tracking table all behave identically — only the framing differs. Errors propagate as normal Node exceptions.

`drizzle-kit generate`, `db:check`, and `db:studio` continue to use the drizzle-kit CLI because their failure modes are observable from interactive use.

### Why no `db:push`?

`drizzle-kit push` skips the migration files entirely and applies schema diffs directly to the DB. That bypasses `__drizzle_migrations` and creates trivial paths to prod drift. We deliberately don't ship it as a script. If you want it for local prototyping, run `bunx drizzle-kit push` manually with full awareness — but commit a real migration before merging.

## Conventions for schema authors

- **One file per table cluster.** `quotes.ts` defines both `quotes` and `quoteItems`; `tasks.ts` defines `tasks` and `userTasks`. Re-export everything from `schema/index.ts`.
- **CHECK constraints are first-class.** Use `check('canonical_name', sql\`...\`)` in the table options. The canonical name should match what `schema.sql` would have generated (`<table>_<column>_check` for inline CHECKs, `chk_<table>_<column>` for the discount/unit type group).
- **Foreign keys go on the column.** `column.references(() => other.id, { onDelete: 'cascade' })`. Cross-file references are fine — Drizzle defers resolution via the callback, so circular file imports work.
- **Type narrowing via `$type<>()`**. Where the column has a CHECK constraint enforcing an enum, also pass `.$type<UnionType>()` so reads from the table are typed precisely (e.g. `discount_type` is `'percentage' | 'currency'` rather than `string`).
